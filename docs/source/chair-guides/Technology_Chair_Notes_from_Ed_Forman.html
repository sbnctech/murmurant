Technology Chair Notes from Ed Forman

November 2025

These are personal notes and lessons learned from my time as Technology Chair. I hope they help future chairs understand that you don't need to be a programmer to manage these systems effectively.

How This System Was Built

I am not a programmer. I haven't written code by typing it since I learned BASIC in the mid-1970s. Everything in the wa-dev repository - all the code, all the documentation - was created using Claude.AI as a resource to implement code and compose documentation.

If I can do this, so can you.

My Best Support Resources: AI Assistants

By far, my best support resources were Claude (claude.ai) and ChatGPT. They are exceptionally strong at:

  • Writing code and documentation
  • Finding bugs and conflicts
  • Explaining things I don't understand
  • Knowing all the systems we use, including Wild Apricot

These AI tools know Wild Apricot well and can help troubleshoot issues faster than traditional support channels.

A Note on Wild Apricot Support:

Wild Apricot's support can be slow, and their agents sometimes provide inaccurate information. There are very few end-to-end support resources for Wild Apricot - most consultants specialize in just one aspect of the platform.

Using AI to Solve Problems

First Stop: The wa-dev Repository

The GitHub repository contains extensive documentation and well-commented code. Start there when researching any issue.

I've found that Claude (claude.ai) and ChatGPT are remarkably capable at fixing problems when you give them:

  • The exact text of error messages
  • A detailed description of what's happening vs. what should happen
  • Relevant code or configuration files from the repository

Claude can rapidly review information from the repository. My common workflow is to simply paste in:

  • Error emails from cron jobs
  • Notes from members describing issues
  • Screenshots or copied text from error screens

Claude Code often fixes problems automatically ("automagically") when given this information.

When to Bring in Professional Help

Time Check: If a problem takes you and Claude/ChatGPT more than an hour or two to resolve, it's time to bring in an experienced programmer. A professional familiar with GitHub and Unix shell commands can typically jump right in - the code is well-documented internally to simplify debugging, and many routines have a debug mode.

Support Resources

Server Issues - Hosting.com

For server-related issues, the hosting.com online support team is excellent. They're patient and can work with you regardless of your technical experience level. Don't hesitate to reach out to them for server problems.

DNS and Email Deliverability - FTI Services

For DNS issues, we worked with FTI Services. They have the specialized expertise needed for complex DNS configurations.

Website: https://www.ftiservices.com/managed-it-services-santa-barbara/

Website Design - Merge Creative

For tuning the look and feel of the site, styling particular elements, or creating new pages that merit high-quality design, I recommend Merge Creative (Alan and Gail). They specialize in Wild Apricot websites, which is important because Wild Apricot has an idiosyncratic web publishing system that would have a long learning curve for a graphic designer unfamiliar with the platform.

In late 2025, we did a website revamp that was supported by Merge Creative.

Important limitation: Their expertise does not extend beyond the four corners of Wild Apricot - they do not support JavaScript or API extensions. For custom code development, you'll need to use AI tools or bring in a programmer.

The 2025 Email Deliverability Crisis

Background:

In 2025, email hosts (Gmail, Yahoo, etc.) significantly tightened their spam standards, causing a deliverability crisis for SBNC. We had to carefully tune our DNS records (SPF, DKIM, DMARC) to meet the new requirements.

The challenge: Wild Apricot runs on shared server infrastructure and cannot modify their DKIM selectors for individual clients. To properly authenticate our email, we had to move our forwarding services from a shared server to a VPS (Virtual Private Server) where we could configure our own selectors.

The trade-off: This increased complexity significantly - we went from a fully managed hosting service to running our own UNIX server. This is why the server documentation and troubleshooting guides exist in the repository.

If you encounter email deliverability problems (messages going to spam, bouncing, etc.), this is a specialized area that may require expert help. The DNS_AND_EMAIL_SETUP.md document in the repository has the current configuration details.

Final Thoughts

The technology should serve the club, not require the club to serve the technology. Everything was built with the goal of making life easier for volunteers, not harder.

Don't be intimidated by what looks like complex systems. With AI assistants and good documentation, you can manage and even improve these systems without being a programmer.

Good luck!
